# Benchmarking API Drift in AI-Generated Quantum Code Across Successive SDK Versions

A reproducible benchmark measuring how LLM-generated Qiskit code degrades across
SDK versions, and whether documentation-augmented prompting can repair it.

---

## Configured Models

Copy `config.py.sample` to `config.py` before running the project:

```bash
cp config.py.sample config.py
```

The real `config.py` is intentionally ignored so you can keep machine-specific
or experiment-specific settings local.

All runnable scripts assume you invoke them from the repository root
(`quantum-api-drift/`). They add that root to `sys.path` so shared helpers in
`lib/` can be imported consistently across prompt preparation, generation,
execution, repair, and analysis.

Configured models live in `config.py` under:
- `GROQ_MODELS`
- `CODEX_MODELS`
- `OPENAI_MODELS`
- `ANTHROPIC_MODELS`
- `OPENROUTER_MODELS`
- `OPENAI_COMPATIBLE_MODELS`

Treat your local `config.py` as the source of truth for the current evaluation set.
The README examples below are illustrative, but the active model list may change
over time as you add, remove, or retune model entries.

## Step 0 -- Provide Your Input Files

### Raw Problems (`data/problems/raw_problems.json`)

```json
[
  {
    "id": "problem_001",
    "description": "Create a quantum circuit for the given integer n_qubits and return it. You must implement this using a function named create_quantum_circuit with the following argument: n_qubits.",
    "entry_point": "create_quantum_circuit",
    "test_call": "result = create_quantum_circuit(3)\nassert result is not None"
  },
  {
    "id": "problem_002",
    "description": "Implement a Bell state circuit using Qiskit and return it. You must implement this using a function named create_bell_state with no arguments.",
    "entry_point": "create_bell_state",
    "test_call": "result = create_bell_state()\nassert result is not None"
  }
]
```

Provide your own problem set following the format above.
You can source these from any quantum coding benchmark (e.g., [Qiskit HumanEval](https://github.com/qiskit-community/qiskit-human-eval)), documentation, or create them manually.

### Migration Notes

Place plain text files in `data/migration_notes/`:

- `v0_to_v1.txt` -- copy the relevant breaking changes from:
  https://quantum.cloud.ibm.com/docs/en/guides/qiskit-1.0

- `v1_to_v2.txt` -- copy the relevant breaking changes from:
  https://quantum.cloud.ibm.com/docs/en/guides/qiskit-2.0

Focus on the sections about circuit construction, transpilation, and backend interfaces.
You do not need the entire guide, just the API change tables and removal notices.

## Step 1 -- Set Up Conda Environments

```bash
chmod +x setup_envs.sh
./setup_envs.sh
```

This creates three environments: `qiskit-v0`, `qiskit-v1`, `qiskit-v2`.

Verify:
```bash
conda run -n qiskit-v0 python -c "import qiskit; print(qiskit.__qiskit_version__)"
conda run -n qiskit-v1 python -c "import qiskit; print(qiskit.__version__)"
conda run -n qiskit-v2 python -c "import qiskit; print(qiskit.__version__)"
```

## Step 3 -- Run Generation

### All Families

```bash
python experiments/run_generation.py
```

This top-level runner dispatches to Groq, Codex, OpenAI, Anthropic, OpenRouter, and generic OpenAI-compatible generation
scripts based on the configured model lists, using shared family/provider logic in
`lib/`. You can also target a single model:

```bash
python experiments/run_generation.py --model kimi_k2
python experiments/run_generation.py --model gpt_5_4_medium
python experiments/run_generation.py --model qwen3_32b
python experiments/run_generation.py --model nemotron_3_super
python experiments/run_generation.py --model gpt_4_1
python experiments/run_generation.py --dry-run
```

Outputs land in `data/generations/{model_label}.json`.

### Groq API

```bash
export GROQ_API_KEY="your_key_here"
export GROQ_BASE_URL="https://api.groq.com/openai/v1"  # optional; this is the default

python experiments/generation/run_generation_groq.py

# Single model only
python experiments/generation/run_generation_groq.py --model kimi_k2

# Dry run (print prompts without calling API)
python experiments/generation/run_generation_groq.py --dry-run
```

Configure models in `GROQ_MODELS` in `config.py`. If a Groq model needs
provider-specific request fields such as `reasoning_effort`, set them in
`extra_body`; those keys are merged into both generation and repair requests.

The script saves results incrementally after every problem, so you can safely
interrupt and resume. Outputs land in `data/generations/{model_label}.json`.

### For Codex CLI

Codex runs via the Codex CLI tool rather than the Groq API and has its own script.

**One-time setup:**
```bash
# Install Node.js 18+ if not already installed, then:
npm install -g @openai/codex
codex --version   # verify installation
```

Authenticate with either option:

- API key:
```bash
export OPENAI_API_KEY="your_openai_key_here"
```
- ChatGPT sign-in / OAuth:
```bash
codex --login
```

The ChatGPT sign-in flow stores credentials locally and creates an API key for
the CLI automatically, so manual key copy-paste is not required.

**Run generation:**
```bash
python experiments/generation/run_generation_codex.py
python experiments/generation/run_generation_codex.py --model gpt_5_4_medium

# Dry run to verify prompts without calling the API
python experiments/generation/run_generation_codex.py --dry-run
```

Output is saved to `data/generations/{model_label}.json` in the same format as Groq
outputs, so the execution, repair, and metrics scripts work on it unchanged.

The script uses `codex exec` in a fresh temporary directory with a read-only
sandbox, so Codex stays non-interactive, does not inspect your repository, and
cannot modify it during generation. Configure Codex runs in `CODEX_MODELS` in
`config.py`, using entries with `id`, `label`, and `reasoning_effort`.

### For OpenAI API

Use this for direct first-party OpenAI API models, separate from Codex CLI and
separate from third-party OpenAI-compatible endpoints.

**One-time setup:**
```bash
export OPENAI_API_KEY="your_openai_key_here"

# Optional custom base URL
export OPENAI_BASE_URL="https://api.openai.com/v1"
```

**Configure which model to run** in `config.py` under `OPENAI_MODELS`.
Model strings must match OpenAI API model IDs exactly. You can also set
`requests_per_minute` per model. If you need provider-specific request options,
set `extra_body`; its keys are merged into both generation and repair requests.

**Run generation:**
```bash
python experiments/generation/run_generation_openai.py
python experiments/generation/run_generation_openai.py --model gpt_4_1
python experiments/generation/run_generation_openai.py --dry-run
```

Output is saved to `data/generations/{model_label}.json` in the same format as
all other model outputs.

### For Claude (Anthropic API)

Claude runs via Anthropic's OpenAI-compatible API and has its own script.

**One-time setup:**
```bash
export ANTHROPIC_API_KEY="your_anthropic_key_here"
```

Output is saved to `data/generations/{model_label}.json` in the same format as
all other model outputs.

### For Generic OpenAI-Compatible Endpoints

Use this when a provider exposes the standard `/chat/completions` API shape and
you want to configure a custom base URL and API key without routing through OpenRouter.

**One-time setup:**
```bash
# Optional shared fallback for models that do not define their own credentials
export OPENAI_COMPATIBLE_API_KEY="your_key_here"
export OPENAI_COMPATIBLE_BASE_URL="https://your-provider.example/v1"
```

**Configure which model to run** in `config.py` under `OPENAI_COMPATIBLE_MODELS`.
Model strings must match whatever your target endpoint expects. Each model can
set its own `api_key`, `api_key_env`, `base_url`, and `base_url_env`, with
fallback to the shared `OPENAI_COMPATIBLE_*` environment variables. You can
also set `requests_per_minute` per model, just like the other providers.
If your endpoint needs provider-specific request options such as NVIDIA's
`reasoning_budget` or `chat_template_kwargs`, set `extra_body` in `config.py`;
its keys are merged into both generation and repair requests. If a provider
needs custom HTTP headers, set `extra_headers` in `config.py`; those headers
are added on top of the default JSON and authorization headers.

**Run generation:**
```bash
python experiments/generation/run_generation_openai_compatible.py
python experiments/generation/run_generation_openai_compatible.py --model your_model_label
python experiments/generation/run_generation_openai_compatible.py --dry-run
```

Output is saved to `data/generations/{model_label}.json` in the same format as
all other model outputs.

## Step 4 -- Run Execution

```bash
# All models
python experiments/run_execution.py

# Single model
python experiments/run_execution.py --model llama_3_3
```

Each snippet is executed in all three Conda environments.
Results are saved to `data/execution/{model_label}_execution.json`.

If Conda is not available, you can substitute Docker:
  Replace the `conda run -n {env}` call in `run_execution.py` with:
  `docker run --rm qiskit-{version} python /tmp/snippet.py`
  and adjust the temp file mounting accordingly.

## Step 5 -- Run Repair Pass

```bash
python experiments/run_repair.py

# Target only drift toward v2
python experiments/run_repair.py --target-version v2

# Single model only
python experiments/run_repair.py --model kimi_k2
python experiments/run_repair.py --model gpt_5_4_medium
python experiments/run_repair.py --model qwen3_32b
python experiments/run_repair.py --model nemotron_3_super
python experiments/run_repair.py --model gpt_4_1
python experiments/run_repair.py --model your_model_label

# Provider-specific entry points
python experiments/repair/run_repair_groq.py
python experiments/repair/run_repair_codex.py
python experiments/repair/run_repair_openai.py
python experiments/repair/run_repair_anthropic.py
python experiments/repair/run_repair_openrouter.py
python experiments/repair/run_repair_openai_compatible.py
```

This re-prompts with migration notes for all snippets that:
- Passed in their target version (version fidelity achieved)
- But failed in at least one newer version (forward drift detected)

Results are saved to `data/repairs/{model_label}_repair.json`.

## Step 6 -- Compute Metrics

```bash
python analysis/compute_metrics.py
```

Prints to stdout:
- Drift matrix per model (3x3 Pass@k rates)
- Error taxonomy (what types of errors cause failures)
- Repair success rates overall, by target version, and by error type

Saves full metrics to `data/analysis/metrics_summary.json`.

## Configuration Reference (`config.py`)

This section summarizes the parts of `config.py` that control the benchmark.
Provider-specific environment variables and run commands are documented in the
provider sections above, so they are not repeated in full here.

### Core Benchmark Settings

| Setting             | Default | Description                                 |
|---------------------|---------|---------------------------------------------|
| `TEMPERATURE`       | `0.8`   | Sampling temperature for generation/repair  |
| `MAX_TOKENS`        | `1024`  | Maximum tokens per model response           |
| `K_SAMPLES`         | `3`     | Samples per prompt for Pass@k evaluation    |
| `EXECUTION_TIMEOUT` | `30`    | Seconds before aborting snippet execution   |

### Paths

These usually do not need to be changed unless you are reorganizing the repo:

- `ROOT` -- repository root resolved from `config.py`
- `PROBLEMS_DIR` -- input problem set directory
- `GENERATIONS_DIR` -- raw generation outputs
- `EXECUTION_RESULTS_DIR` -- execution result files
- `REPAIRS_DIR` -- repair result files
- `ANALYSIS_DIR` -- computed metric summaries
- `MIGRATION_DIR` -- migration note text files
- `ANCHORED_FILE` -- generated version-anchored prompt file

### Qiskit Versions

- `QISKIT_VERSIONS` maps each benchmark version key (`v0`, `v1`, `v2`) to:
  an `anchor_prefix` used when building prompts and a `conda_env` used during execution.
- If you change version labels, prompt wording, or environment names, update this
  mapping so prompt preparation, execution, repair, and metrics stay aligned.

### Provider Configuration

- `GROQ_MODELS`, `CODEX_MODELS`, `OPENAI_MODELS`, `ANTHROPIC_MODELS`,
  `OPENROUTER_MODELS`, and `OPENAI_COMPATIBLE_MODELS` define which models are active.
- Each model entry should include a stable `id` and a local `label`.
- Optional per-model fields used by the runners include:
  `requests_per_minute`, `reasoning_effort`, `extra_body`, `extra_headers`,
  `api_key`, `api_key_env`, `base_url`, and `base_url_env`.
- Provider credentials and base URLs are read from environment-backed settings in
  `config.py`; use the earlier provider sections for the expected variables.
- `ALL_MODELS` is derived automatically from the provider-specific model lists and
  does not need to be edited directly.

## Limitations

- Semantic correctness (whether circuits are mathematically correct) is out of scope.
  Only syntactic/API-level validity is measured.
- Conda environments offer slightly less isolation than Docker. Results are still
  valid as long as environments are correctly pinned.
